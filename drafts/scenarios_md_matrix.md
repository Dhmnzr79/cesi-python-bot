# Реальные сценарии по `md/` (простой вид)

> Статусы в этом документе:
> - `✅ Внедрено` — сценарий покрыт текущей логикой в коде.
> - `🟡 Частично` — часть поведения есть, часть остается целевой.
> - `⛔ Не внедрено` — отдельной целевой ветки в коде пока нет.

## Общий принцип работы с `md`

1. **Два слота** под «развитие текущей темы» сначала заполняем только содержимым **этого же** документа: **`suggest_h3`** (переходы между разделами внутри файла) и **`video_key`** (если задано). Это не мосты наружу и не ситуация.
2. **На первом контентном ответе** по `doc_id`, если есть **`video_key`**, у **видео приоритет**: оно выставляется раньше остальных доп. элементов (в паре с одним followup, если очередь `suggest_h3` не пуста — см. `policy.py`).
3. **`suggest_refs`** (мост на другой `.md`) показываем **только когда тема по этому документу исчерпана** для слотов: нет доступных followup из `suggest_h3` по правилам покрытия / счётчика (`topic_exhausted` в policy).
4. **`situation`** (`situation_allowed: true`): **пока только в материалах по имплантации**, для теста сценария; в клинике, врачах, ортодонтии, протезировании, лечении и т.п. **`situation_allowed` не включаем**, пока явно не расширим правило.

Ниже — разбор по фактическим комбинациям YAML. Поведение после чанка — как **обычно** в коде (`policy.py` + `ux_builder.py`), если **нет** `low_score`, **нет** активного lead-flow и **нет** `situation_pending`. Детали слотов — в **`scenario_2.md`**.

# Механика: JSON и md

**Каталог (JSON)** — список всех услуг: название, алиасы, флаги, опционально `facts`, ссылка на md (`md_entry_ref`), ключ цены (`price_key`). Неактивные позиции не матчим. 

**Прайс (JSON)** — только те `price_key`, для которых есть цифра; остальные услуги в каталоге могут быть без строки в прайсе. Редко: `price_ref` — цена/текст из другого места (в т.ч. кусок md).

**md** — длинные тексты и сценарные элементы (разделы, кнопки, видео и т.д.).

**Порядок обработки вопроса**

1. Ценовой интент → матч услуги по каталогу; дальше прайс / `price_ref` / ответ «про цену» без цифры; слабый матч → уточнение, не выдумываем услугу.

2. Не цена (или ценовая ветка «other») → снова матч по каталогу с порогом уверенности:  
   - есть `md_entry_ref` и файл → ответ из md (вход с `#korotko`);  
   - нет md, но есть `facts` → короткая карточка из JSON;  
   - матча нет / режим `none` → обычный поиск по корпусу md (клиника, врачи, всё что не сузилось до одной услуги).


## Роль JSON при запросе услуги

JSON:
- используется для определения услуги
- используется как точка входа в md

Дальше:
- вся логика = md + YAML + policy

## Сценарий: Услуга с md + цена есть в prices.json

Пользователь спросил про услугу, но НЕ спросил цену.

### Что происходит
- Бот находит услугу в JSON-каталоге
- Видит, что по услуге есть md
- Переходит в md
- Дальше работает стандартный md-сценарий

### Что с ценой
- Цена НЕ показывается автоматически
- Цена хранится в фоне как доступная информация
- Цена показывается только если:
  - пользователь прямо спросил стоимость
  - или в настройке услуги указано `price_display: always`.

  Идея зачем нужен price_display: always. На топовых услугах у клиники ценник выше среднего. И цена в первом же сообщений может отпугнуть человека. Поэтому есть идея на дорогие услуги цену давать по запросу.

### Что показываем
- Ответ по md
- Followup / video / situation / suggest_refs — по YAML
- CTA — по YAML

## Сценарий: Цена после вопроса про услугу

Тип запроса: цена  
Услуга в текущем сообщении: не указана  
Услуга в предыдущем контексте: есть  

Пример:
Пользователь: а КТ делаете?  
Бот: отвечает про КТ  
Пользователь: а сколько стоит?

### Что происходит
- Бот видит ценовой интент
- Не ищет услугу только в новом сообщении
- Берёт последнюю определённую услугу из контекста диалога
- Проверяет цену по этой услуге в prices.json

### Что показываем

Если цена есть:
- Ответ
- Цена
- CTA — по YAML

Если цены нет:
- Ответ, что точную стоимость лучше уточнить у администратора
- CTA — по YAML / lead-flow

### Важно
- Бот НЕ должен спрашивать “какую услугу вы имеете в виду”, если в предыдущем ходе уже была определена услуга
- Последняя услуга должна храниться в сессии как активный контекст

---

## Правило контекста для цены

Если пользователь спрашивает “сколько стоит?”, “а цена?”, “а сколько?” без названия услуги — бот сначала проверяет последнюю услугу в сессии.

Если последняя услуга есть → отвечает по ней.  
Если последней услуги нет → просит уточнить услугу / переводит на администратора.

## Правило матчинга услуги

Бот не должен требовать точного совпадения названия услуги.

Матч должен происходить через:
- алиасы
- упрощённые формулировки
- базовые слова (например: "отбеливание", "имплант", "брекеты")

Точное название из базы — НЕ обязательно.

## ⛔ Сценарий: Спам / флуд / нерелевантный запрос

Тип запроса: спам / оффтоп / мусор  
Примеры:
- набор символов: "asdfgh", "123123"
- ругань
- не по теме (не стоматология)

### Что происходит
- Для очевидного мусора (символы/шум/повторы) срабатывает быстрый pre-check  
- Pre-check не запускает handoff LLM, retrieval и answer LLM  
- Для неочевидных случаев включается ранний handoff LLM-filter  
- При label=handoff запрос не идёт в retrieval и answer LLM  

### Что показываем
- Для очевидного мусора: короткий ответ-заглушка, без CTA  
- Для handoff по LLM: единый handoff-ответ; CTA зависит от reason  
- Для spam/offtopic/prompt_injection/acute_symptom/medical_advice/legal — без CTA

## ⛔ Сценарий: Частые осмысленные вопросы (перегруз / сжигание токенов)

Тип запроса: релевантный  
Качество: нормальные вопросы  
Поведение: много подряд сообщений за короткое время  

### Что происходит
- Бот считает общее количество сообщений в сессии

### Что показываем

- Мягкий redirect-ответ  
- CTA: "Связаться с администратором" (lead)
