# Виджет чата — roadmap и ТЗ (сверка с ботом)

Источник правды по **именам полей и веткам** API: код (`app.py`, `flow_handlers.py`, `policy.py`, `ux_builder.py`, `session.py`, `chunk_responder.py`). Продуктовые требования к UI/UX ниже совпадают с `work_info/temp.md`: это **обязательный объём к проду**, а не «опции». Слоты и приоритеты — `work_info/scenario_2.md`.

Сокращать при правках имеет смысл только **повторы и воду** (длинные цитаты MDN/Material и т.п.), а не выкидывать фичи.

---

## 1. Продукт и принцип

Конверсионный мессенджер: быстрый вход в тему, followup, видео в нужный момент, situation, CTA, lead-flow. Фронт — **renderer** по payload `/ask`, без дублирования policy и без локальных эвристик вроде «если пользователь написал „да“, открыть лид» — короткие ответы и редиректы обрабатывает бэкенд (`last_bot_action`, `handle_flows`).

Стриминг текста — **целевое требование к продукту**; транспорт на бэке может включаться позже, но фронт и контракт под стрим закладываются (§9).

---

## 2. Shell и состояния UI

Один DOM-shell (не три отдельных «окна»), морфинг из anchor внизу: resize/layout, без slide-in с краёв и без резкого появления без связи с launcher.

**Геометрия:** `panel` — основной режим; на mobile почти full-screen sheet, на desktop компактная ширина. `expanded` — шире/выше (кнопка в header, длинный ответ, видео, lead). **Header** целиком внутри виджета: аватар, имя, `onlineLabel`, expand, close/minimize.

**Shell:** `collapsed` | `panel` | `expanded`.

**Launcher (collapsed):** аватар, имя, **онлайн-лейбл** (`onlineLabel`), **unread-dot** при новом сообщении/призыве, лёгкая тень/радиус, hover/tap, не перекрывать контент сайта; по клику раскрывается **тот же** shell.

**Наложенные состояния (явная модель, не россыпь флагов):**  
`loading`, `typing`, **`streaming`** (когда включён поток), **`video_open`**, сценарии **`situation_pending`** / **lead** (имя / телефон / финальный экран успеха), **`error`**, **`offline`**, при необходимости `retrying`.

Пока ответ не пришёл: **`loading` + typing** (один индикатор на один активный запрос). После появления стрима — см. §9.

---

## 3. Лента (feed)

Спокойные паттерны блоков, не случайный микс: welcome / сообщения user+bot / строка typing / при стриме — дописывание в активном пузыре / **compact video row** / secondary actions / CTA / карточка situation / карточка lead / карточка ошибки.

Логика «что показывать» по слотам — из payload и `scenario_2.md`; не перегружать экран — следовать policy (`meta.policy_decision`, исчерпание темы и т.д.).

---

## 4. Иерархия под ответом и рендер по режимам

1. Текст `answer`.  
2. **Secondary-группа** (одна визуальная семья: общий радиус/отступы/hover/focus): chips из **`meta.followups`** + chip «ситуация» при `situation.show && mode === "normal"` + **video media row** при `video.key` (чуть выше chip, микро-аватар врача слева при наличии в конфиге/каталоге, play справа) — без автоплея и без старта от скролла; только явный play.  
3. **`quick_replies`** (suggest_ref) — компактно, не меню; не подсвечивать уже пройденные подтемы как «новые» (UX-отметка «уже смотрели» — по правилам продукта).  
4. **`cta`** — отдельный primary слой: заметнее secondary, иконка календаря, отступ сверху, full-width или почти; не в одну строку с chips.

**Situation:** `show=false` — нет блока; `show=true, mode=normal` — только chip; `show=true, mode=pending` — карточка ввода + **«Назад к диалогу»** (`situation_action: "back"`).

**Lead:** только шаги воронки; обычные followup / video / situation / `quick_replies` / CTA в эти моменты **не смешивать** с основным экраном действий.

---

## 5. `POST /ask` — тело запроса

Валидный `client_id`.

| Поле | Назначение |
|------|------------|
| `q` | Текст; сброс: `/reset` или `/новая` |
| `ref` | `file.md#anchor` |
| `sid` | Сессия |
| `client_id` | Тенант |
| `cta_action` | `"lead"` — нажата запись |
| `situation_action` | `"start"` / `"back"` |
| `action` | `"situation"` — как старт ситуации |

Короткие «да» / выбор из кнопок — только через отправку текста или ref так, как ожидает бэкенд; **не** подменять бэкенд локальной логикой.

---

## 6. Ответ `/ask` — структура (как в коде)

| Корень | Смысл |
|--------|--------|
| `answer` | Текст |
| `quick_replies` | suggest_ref: `{ label, ref }` |
| `cta` | `{ text, action }` или `null` (запись: обычно `action: "lead"`) |
| `video` | **`{ "key": "..." }`** или `null` — URL/title с API **не приходят** |
| `situation` | `{ show, mode: "normal" \| "pending" }` |
| `offer` | Сейчас по сути не используется |
| `meta` | в т.ч. **`followups`** (не корень JSON!), `sid`, `client_id`, `policy_decision`, `lead_flow`, `situation_collect`, флаги сценария, `turn_count`, … |

**Важно:** подтемы — **`meta.followups`**, не корневое `followups`. Лимиты: `normalize_policy_payload` (в т.ч. `meta.ui_dropped`).

---

## 7. Lead-flow (бэкенд)

Имя → телефон → **отправка** через `handle_lead` внутри `/ask`; отдельного API-шага «подтвердите имя и телефон» в боте **нет** — на фронте после успеха показывается **финальный экран/сообщение успеха** (и при ошибке — `meta.lead_error`). См. §16 про формулировку «шаг 3 подтверждение» в старых текстах.

Телефон: `type="tel"`, при необходимости `inputmode` / `autocomplete`; валидация формата на сервере.

---

## 8. Situation

Как в §4; после отправки текста ситуации бэкенд переводит в lead — фронт **не** показывает «ещё один умный ответ» по ситуации, а переключается в воронку.

---

## 9. Стриминг и живая подача ответа

**Цель продукта:** typing → (при наличии потока) first token → плавное дописывание в **одном** активном сообщении → по завершении — показ actions под ответом. Технически: не ререндерить всю ленту на каждый токен, не ломать скролл, не несколько typing одновременно.

**Сейчас в бэке:** один JSON без SSE. **Фронт:** один слой API (`fetch` → JSON сейчас; позже EventSource/stream). **Бэк (отдельная задача):** события вроде `token` | `final` | `error`, итоговый JSON как §6.

**A11y:** live region / статус для потоковых обновлений.

---

## 10. Видео

Обязательный **UX** (источник `key` — API): не голубая ссылка по умолчанию, не огромный плакат в ленте, не autoplay, не старт от случайного касания при скролле; compact row + явный play; по play — `video_open` (inline или лёгкий sheet/overlay). Подпись/заголовок для a11y — из **каталога по `key`** или после расширения API (§16).

---

## 11. Ошибки и сеть

Явные состояния/сообщения: сеть, таймаут, **retry**, **offline**, при повторе — `retrying`. Не терять текст ввода, не ломать историю при сбое.

---

## 12. Сессия, персист, конфиг

На сайте сохранять: открыт/закрыт, история (в рамках выбранной стратегии), **`sid`**, **черновик ввода** в composer, состояние lead при обрыве; при переходе услуга → врач не сбрасывать чат в пустой без причины.

**Конфиг (один объект):** `apiBaseUrl`, `client_id`, `botName`, `avatarUrl`, **`onlineLabel`**, `welcomeText`, **`starterPrompts`** (2–3 кнопки на старте: конфигурируемые, **исчезают после начала диалога**, не заменяют свободный ввод), опционально контекст страницы для приветствия, **`themeTokens`** / CSS variables, задержки teaser (`autoOpenDelay` и т.д. — имена по месту реализации). Персонализация только из конфига, без «угадывания» интента на фронте.

---

## 13. Composer

`textarea` 1–4 строки; **Enter** — отправить, **Shift+Enter** — новая строка; кнопка отправки активна только при непустом тексте; на время запроса блокировать повторную отправку (анти-спам).

---

## 14. Motion, визуал, вёрстка

**Motion:** в основном resize, opacity, тень, лёгкий scale; микроподъём нового сообщения; лёгкий pulse typing; **без** slide с краёв, bounce, агрессивного spring. **`prefers-reduced-motion`** — ослаблять несущественное движение.

**CSS:** токены (spacing, radius, shadow, motion), без магических чисел по всему коду; состояния hover/focus/pressed/disabled.

**HTML:** нативные `<button>`, `<header>`, нормальные поля ввода; не `div` как кнопка.

**Визуал:** спокойный messenger, мягкие края, лёгкая глубина, без перегруза и «рекламного» попапа.

---

## 15. A11y (прод)

Нативные кнопки; при открытии shell — фокус внутрь, при закрытии — **возврат на launcher**; полная клавиатура; **Escape** закрывает overlay, если слой модальный; видимый focus ring; понятные имена; для диалогового паттерна — корректная семантика (`dialog` при оверлее).

---

## 16. Безопасность (фронт)

Не хранить секреты; не светить debug token; не подставлять произвольный HTML из `answer` без санитизации; в проде не хардкодить только localhost; `client_id` — из конфигурации окружения.

---

## 17. Архитектура кода (ментальная модель)

Один **ApiClient**; не размазывать `fetch` по кнопкам. Слои: shell, launcher, panel, header, feed, composer; блоки сообщений, typing, streaming message, secondary group, chips followup/situation, video row, primary CTA, situation card, lead steps, success/error; store сессии + конфиг + токены темы.

**Не** строить в рамках этого бота: microfrontend без нужды, отдельный rules engine на фронте, «мультитенант-платформу» сверх уже существующего `client_id`, drag-resize и т.п.

---

## 18. Roadmap (порядок работ)

Локальная проверка UI: **`static/widget-test.html`**. Наглядный сырой JSON и retrieval без смешения с виджетом: **`static/debug/ask-inspector.html`** (папку `static/debug/` можно удалить целиком).

1. Shell + morphing + header + composer.  
2. Feed + welcome + starter prompts.  
3. Маппинг `/ask` §6 + иерархия §4.  
4. Спецрежимы situation / lead + ошибки/офлайн §11.  
5. Видео row + `video_open` §10.  
6. Персист §12.  
7. Стриминг §9 + бэкенд.  
8. Teaser §19 + прод-полировка §14–15.

---

## 19. Teaser (прод, обязательно)

Nonmodal карточка над launcher, визуально связана с ним; **один раз за сессию**; задержка после бездействия из конфига; не показывать, если чат уже открывали; закрытие крестиком или автоскрытие (тайминги в конфиге); клик открывает тот же shell (launcher → panel); копирайт рабочее/нерабочее время — из конфига клиники; на mobile — осторожнее с таймингом (отдельный конфиг допустим).

Неправильно: агрессивный центральный модал, перекрытие launcher целиком, повтор teaser после закрытия в той же сессии.

---

## 20. Критерии готовности (кратко)

Один объект shell; визуальная система chips / video row / CTA; спецрежимы не смешаны с обычным экраном; typing/стрим не ломают layout; a11y и reduced motion; телефон удобен на mobile; сеть не стирает чат; сессия переживает навигацию; онлайн-лейбл и teaser в проде.

---

## 21. Синхронизация с другими артефактами

Устарело в старых заметках: корневые `followups`, `video: { title, url }` с бэка, `cta` с `booking` если в контенте везде `lead` — смотреть код и этот файл.

---

## 22. Спорные моменты и расхождения с бэкендом

| Тема | Как в документах-идеях (temp) | Как в боте сейчас |
|------|------------------------------|-------------------|
| Lead «шаг 3 — подтверждение» | Отдельный экран подтверждения данных | После телефона сразу submit и ответ успех/ошибка; отдельного шага подтверждения **нет** — на фронте достаточно **экрана успеха** (и retry при ошибке). |
| Стриминг | «Обязателен» как UX | Бэкенд отдаёт цельный JSON; стрим — **целевое** поведение, требует доработки API §9. |
| `video` | title, poster, врач | С API только `key`; превью/текст/URL — **каталог на фронте или будущее поле API** §10. |
| `offer` | Может фигурировать в типах | В ответах по сути не используется — не опираться до появления логики. |
| «Подтему уже открывали» | Визуальная метка | Бэк отдаёт `covered` через policy/meta косвенно; **как именно подсвечивать** — продуктовое правило поверх payload (не выдумывать refs заново). |

Если появятся новые поля в JSON — обновить §6 и эту таблицу.
