## YAML-first slot policy (финальная версия для 2.0)

### 1. Общий принцип
YAML — это источник правды по теме.

Если элемент явно разрешён в YAML, policy не должна спорить с этим решением.
Policy только:
- проверяет текущий режим,
- убирает уже показанное,
- раскладывает доступные элементы по слотам по приоритету.

---

### 2. Отдельное поле `situation_mode: early|normal` не вводим
Для 2.0 это лишняя бюрократия.

Если в документе стоит:
- `situation_allowed: true` — значит кнопку ситуации можно показывать;
- есть `video_key` — значит видео нужно показывать;
- есть `suggest_refs` — значит эти переходы нужно показывать;
- есть `suggest_h3` — значит по ним нужно строить внутреннюю навигацию.

---

### 3. Слоты экрана и CTA
Для обычного контентного ответа используем:

- **1 слот под CTA**
- **2 слота под дополнительные элементы**

CTA:
- **первый ответ темы — без CTA** (пользователь ещё “холодный”);
- CTA показывается начиная **со второго осмысленного ответа** по этой теме;
- неважно, как пользователь пришёл к этому ответу — по followup или свободным вопросом.

CTA не показывается:
- в `situation_pending`;
- внутри активного lead-flow;
- при явном booking intent: в этом случае диалог сразу переводится в lead-flow (имя → телефон), а CTA не нужен.

---

### 4. Что считается дополнительными элементами
Дополнительные элементы:
- `followup` — из `suggest_h3`
- `video` — из `video_key`
- `situation` — если `situation_allowed: true`
- `suggest_ref` — из `suggest_refs`

---

### 5. Жёсткий приоритет дополнительных элементов
Если свободные слоты есть, заполняем их по порядку:

1. `followup`
2. `video`
3. `situation`
4. `suggest_ref`

Это значит:
- сначала раскрываем текущую тему,
- потом показываем конверсионное видео по теме,
- потом даём персональный шаг,
- потом показываем мостик в соседнюю тему.

---

### 6. Правило показа
Если элемент:
- разрешён YAML,
- ещё не был показан или не исчерпан,
- не конфликтует с активным режимом,
- и под него есть свободный слот,

то он **должен быть показан**.

Не используем дополнительные “умные” расчёты вида:
- “дозрел ли пользователь”
- “важная ли тема по субъективной логике”
- “может, ref лучше показать позже”

Если элемент есть и есть слот — показываем по иерархии.

---

### 7. Что считаем недоступным для повторного показа
Не показываем повторно:
- уже показанное `video` в этой теме;
- уже предложенную `situation` в этой теме;
- уже покрытые `followup`;
- уже использованные или неактуальные `suggest_ref`, если есть такое ограничение в реализации.

---

### 8. Активные режимы имеют приоритет над slot-policy
Если включён специальный режим, обычная slot-policy временно не работает.

#### `situation_pending`
Показываем только:
- приглашение описать ситуацию;
- экран ввода ситуации;
- кнопку `Назад к диалогу`.

Фронт рендерит блок `situation` строго так:

- `show = false, mode = "normal"` → блока нет;
- `show = true, mode = "normal"` → только кнопка «Рассказать о ситуации»;
- `show = true, mode = "pending"` → только экран ввода + кнопка «Назад к диалогу».

Режим `mode = "pending"` полностью переопределяет видимость обычной кнопки, чтобы не было смешения normal/pending.

#### lead-flow
Показываем только шаги воронки:
- имя;
- телефон;
- финальное подтверждение.

Обычные `followup`, `video`, `situation`, `suggest_ref`, `CTA` в этих режимах **не показываются**.

---

### 9. Как считать тему раскрытой
Тема раскрывается через элементы текущей темы:
- `followup`
- `video`

Пока есть доступные `followup` и/или ещё не было показано обязательное `video`, они выше `situation`.

Как только слоты после элементов текущей темы освобождаются:
- в следующий свободный слот ставим `situation`;
- потом в следующий свободный слот ставим `suggest_ref`.

Счётчики:
- `doc_turn_count` учитывает только **обычные контентные ответы** по данной теме;
- не входят: экраны `situation_pending`, шаги lead-flow, служебный back и другие спецэкраны без контента по doc_id.

---

### 10. Видео
Если у темы есть `video_key`, видео:
- показывается **один раз** в рамках темы;
- имеет приоритет выше `suggest_ref`;
- не должно теряться из-за длинного прокликивания темы.

Момент фиксации:
- `video_shown = true` считается в тот ответ, где видео попало в payload (а не по клику).

Если у темы есть и `followup`, и `video`, на **первом** экране темы показываем:
- `1 followup`;
- `video`;
- без CTA (CTA включается со второго ответа).

---

### 11. Situation
Если `situation_allowed: true`, кнопку ситуации показываем не по субъективному правилу “рано/не рано”, а по slot-policy.

То есть:
- как только до неё доходит очередь по приоритету,
- и есть свободный слот,
- показываем её обязательно.

После клика:
- включаем `situation_pending`;
- показываем приглашение описать ситуацию;
- показываем кнопку `Назад к диалогу`.

Поведение:
- следующее текстовое сообщение всегда считаем описанием ситуации;
- после получения текста:
  - сохраняем note;
  - выключаем `situation_pending`;
  - сразу переводим в lead-flow: имя → телефон.

Обратный переход:
- по нажатию «Назад к диалогу»:
  - выключаем `situation_pending`;
  - возвращаемся к обычному ответу по теме без повторного показа кнопки ситуации в этом же ответе;
  - так как `situation_offered = true` уже стоит с первого показа, кнопка ситуации в этой теме больше не повторяется.

---

### 12. Suggest_ref
`suggest_ref` — всегда последний по приоритету.

Если он есть в YAML и до него дошла очередь по слотам, он должен быть показан.
Не нужно специально задерживать его “на потом”, если слот свободен.

За один экран желательно показывать не больше 1 `suggest_ref`.

---

### 13. Короткая формула для Cursor
YAML определяет состав допустимых элементов.
Policy не спорит с YAML.
Policy только:
- исключает активные спецрежимы,
- убирает уже показанное,
- заполняет 2 слота по приоритету:

`followup -> video -> situation -> suggest_ref`

---

### 14. Сверка сценариев с текущей реализацией (as-is)

Этот блок фиксирует, как сценарии работают в коде сейчас, чтобы фронт и документация были синхронизированы.

#### 14.1 Что реализовано стабильно

- Спецрежимы действительно имеют приоритет:
  - `lead_flow` перекрывает обычный контентный экран;
  - `situation_pending` перекрывает обычный экран и ожидает ввод ситуации.
- Короткое “да” обрабатывается детерминированно по `last_bot_action`:
  - после CTA -> запуск lead-flow;
  - после situation-offer -> переход в `situation_pending`.
- После текста ситуации бот переводит пользователя в lead-flow (шаг имени).
- При `situation_action=back` есть восстановление последнего контентного UI-снимка.

#### 14.2 Важные уточнения по точности сценариев

- Источник для followup/ref/video/situation — frontmatter в `md/*.md` (YAML внутри markdown), а не отдельные `.yaml` файлы.
- Политика показа работает поверх уже выбранного чанка; поэтому итоговая сценарная точность напрямую зависит от качества chunk selection (rewrite + aliases + broad-query + rerank + low-score guard).
- Для коротких и неоднозначных реплик основной риск — не в policy, а в неверном попадании в тему до policy-этапа.

#### 14.3 Что считать “на перспективу”

- **Мультиклиентность:** пока описывать как roadmap-направление (полная tenant-изоляция данных/индексов/сессий), не как завершенную часть сценарного ядра.
- **Видео:** считать тестовым/вспомогательным каналом. Для текущего этапа сценарной и retrieval-точности видео не является критическим блоком.

#### 14.4 Приоритет для фронта на текущем этапе

Фронту в первую очередь опираться на стабильные сценарные флаги:
- `situation.show` + `situation.mode`,
- `cta` (наличие/отсутствие),
- `quick_replies`,
- `meta.followups`,
- `meta.policy_decision` (для диагностики).

Главный фокус этапа: корректно отрисовать сценарные состояния и не ломать детерминированные переходы “CTA/ситуация/лид”, пока мы донастраиваем точность retrieval.
