## Этап 1 — рефакторинг

| № | Задача (док) | Статус | Детали |
|---|----------------|--------|--------|
| 1 | Разнести `app.py` по модулям по структуре §10 | **~100%** | Реализованы: `config.py`, `session.py`, `llm.py`, `retriever.py`, `policy.py`, `ux_builder.py`, `lead_service.py`; в `app.py` в основном роуты и обвязка. В репозитории пока нет `clients/clients.json` из схемы §10 — относится к продуктовой/этап‑3 конфигурации. |
| 2 | Объединить `SESSION_STATE` и `SESS` в один `session.py` | **100%** | Единая модель состояния; позже перенесена в SQLite (этап 2). |
| 3 | Схлопнуть четыре блока сборки ответа в `build_response()` | **100%** | Одна точка: `ux_builder.build_ask_response()`; в `app.py` — `finalize_ask()` (policy + фиксация `last_*`). |
| 4 | Добавить `client_id` во все слои | **Частично** | **Есть:** `resolve_client_id`, 403 при неизвестном id, `DEFAULT_CLIENT_ID`, поле в `meta`, проброс в сборку ответа. **Нет:** изолированные индексы/корпус по клиенту, составной ключ сессии `(client_id, sid)`, использование `client_id` внутри retrieval как отдельного тенанта — это пересекается с **этапом 3, п.13**. |

## Этап 2 — ядро

| № | Задача (док) | Статус | Детали |
|---|----------------|--------|--------|
| 5 | Вынести policy layer в `policy.py` с **таблицей правил** | **Частично** | **Есть:** приоритет контакты/цены, `booking_intent`, снятие CTA при `turn_count < 2` (кроме явной записи), заготовка `lead_intent`, ответы с `meta.low_score` не режутся по CTA. **Нет:** явная упорядоченная таблица из §5 (10 правил): воронка, подтемы/мостики, «ситуация» после N ходов, смена темы, handoff, дефолт из YAML и т.д. |
| 6 | `last_bot_action`, `last_offer_type`, `last_presented_buttons` в сессии | **100%** | Поля в JSON сессии в SQLite; обновление в `record_last_bot_payload()` после финального payload. |
| 7 | Broad query detector и low-score fallback | **~100%** | `broad_query_detect`, `prefer_overview_if_broad` в `retriever.py`; порог `LOW_SCORE_THRESHOLD`, ответ и логика отсечения в `app.py` / `ux_builder.low_score_response`. Fallback формально размазан по модулям (не только `retriever.py`), по поведению соответствует документу. |
| 8 | SQLite вместо `SESS = {}` и JSONL | **~100%** | Сессии в `data/bot.db` (путь задаётся `SQLITE_PATH`). Исторический in‑memory словарь убран. **Примечание:** в рефакторенном коде не было отдельного JSONL‑хранилища **сессий** в приложении; каталог `data/sessions/` мог использоваться вручную/в старых сценариях — автоматической миграции в SQLite не делалось. |
| 9 | Structured outputs (JSON mode) для OpenAI | **Частично** | **Есть** для основного чата в `llm.py` (`response_format=json_object`, разбор `{"answer": ...}`), переключатель `CHAT_JSON_MODE`. **Нет** для `llm_rerank` в `retriever.py` (по-прежнему разбор свободного текста). Расширенная схема (несколько полей через strict schema) не внедрялась. |

## Открытые вопросы

1. **Мультитенант (п.4 / этап 3 п.13):** когда появится второй клиент — сразу ли разводить `data/{client_id}/` для корпуса и эмбеддингов и ключ сессии с `client_id`?
2. **Policy §5:** планировать ли следующий инкремент как явный список правил с порядком применения и тестами на каждое?
3. **Старые `data/sessions/*.json`:** нужна ли одноразовая миграция в SQLite или формат больше не используется?
4. **Rerank:** вводить ли JSON/structured output для выбора номера чанка, чтобы убрать хрупкий парсинг цифры из текста?
