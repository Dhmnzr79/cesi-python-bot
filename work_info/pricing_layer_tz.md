# ТЗ: ценовой слой бота (без правок кода на этом этапе)

## Цель
Убрать ошибки в ответах по ценам и услугам за счет детерминированного источника цифр и управляемой маршрутизации:
- цифры цен не генерируются LLM;
- бот не путает `price_lookup` и `price_concern`;
- выбор услуги и ценового сценария прозрачен в логах.

## Область работ
В рамках задачи проектируется целевой контур (подготовка данных + правила), без внесения изменений в код на текущем шаге.

---

## 1) Целевой принцип решения

1. Цены хранятся централизованно в `prices.json` (для простых услуг) и в ограниченном наборе ценовых MD-доков (для сложных пакетов).
2. Остальные MD-документы не содержат чисел цен.
3. Маршрутизация идет по интентам:
   - `price_lookup` — запрос конкретной цены/стоимости;
   - `price_concern` — возражения/страх по цене;
   - `other` — обычный информационный запрос.
4. Для `price_lookup` выбирается конкретная услуга из каталога с учетом confidence.
5. При низкой уверенности — уточняющий вопрос, а не “угадывание”.

---

## 2) Целевая структура данных

Для каждого клиента:

- `clients/<client_id>/service_catalog.json`
- `clients/<client_id>/prices.json`

Контент:

- `md/<client_id>/...` (основные документы без цен)
- `md/<client_id>/*__pricing__*.md` (сложные ценовые сценарии, где допустимы цифры)
- `md/<client_id>/*__faq__cost*.md` (обработка возражений по цене, без цифр)

---

## 3) Требования к `service_catalog.json`

Для каждой услуги должны быть поля:

- `title` — название услуги;
- `aliases` — реальные пользовательские формулировки;
- `active` — флаг включения;
- `response_mode` — `card` или `md`;
- `md_entry_ref` — входной контент услуги (если нужен MD);
- `price_key` — ключ в `prices.json` (для простых цен);
- `price_ref` — ссылка на ценовой MD (для сложных цен);
- `concern_ref` — ссылка на FAQ по дороговизне именно этой услуги;
- `price_display` — `always` или `on_request`;
- `suggest_refs` — до 1 элемента для card-ответа.

Правила:
- если `price_ref` задан, при `price_lookup` приоритет у `price_ref`;
- если `price_key` задан и `price_ref` пуст, цена берется из `prices.json`;
- `concern_ref` обязателен для услуг, где важен сценарий “дорого/сомнения”.

---

## 4) Требования к `prices.json`

Единый источник цифр, поддержка типов:
- `fixed`
- `from`
- `range`

Обязательные поля для каждой позиции:
- `name`
- `price_type`
- `currency`
- `value` / `value_min` + `value_max` (в зависимости от `price_type`)
- `note` (nullable)

Правило рендера:
- цена форматируется только кодом;
- LLM не меняет и не пересчитывает цифры.

---

## 5) Интент-слой (требования к логике)

Целевой порядок:

1. Определение интента (`price_lookup | price_concern | other`) с приоритетом точности.
2. Матч услуги в каталоге с оценкой уверенности (`match_score`).
3. Решение:
   - высокий `match_score` -> целевой сценарий;
   - низкий `match_score` -> уточнение услуги;
   - `price_concern` -> переход в `concern_ref` конкретной услуги.

Важно:
- для интента допустим детерминированный regex (без отдельного confidence), если паттерны узкие и проверенные;
- confidence-контроль обязателен на этапе матча услуги (`match_score`), а не на самом факте срабатывания интента;
- избежать глобального `price_concern`-ref “на все случаи”.

---

## 6) Policy/UX требования для ценового слоя

Для источника `catalog/card`:
- не применять обычную slot-policy контентной темы в полном объеме;
- ответ краткий, фактический;
- `suggest_refs` не более 1;
- CTA — по согласованному правилу (включать/не включать отдельно от контентного сценария).

Для `price_lookup`:
- если есть точная цена -> выдать цену в явном виде;
- если диапазон/“от” -> выдать строго в этом формате;
- если цена отсутствует -> честно сообщить и предложить уточнение.

Для `price_concern`:
- эмпатия + объяснение факторов стоимости;
- без выдуманных цифр.

---

## 7) Логирование (обязательный минимум)

В событиях должны фиксироваться:
- `intent`;
- `intent_confidence` (если есть);
- `matched_service_id`;
- `match_score`;
- `route_source` (`catalog` / `price_ref` / `prices_json` / `md`);
- `price_key` / `price_ref` (если использованы);
- `fallback_reason` (если был переход к уточнению/общему ответу).

Цель: дебаг ошибок “не та услуга / не та цена” без ручного расследования.

---

## 8) Критерии приемки

1. На запросы вида “сколько стоит <услуга>” бот возвращает цену из целевого источника (не выдумывает).
2. На возражения “дорого/почему так” бот не уходит в прайс-выдачу, а отвечает через `concern_ref`.
3. При неоднозначной услуге бот задает уточняющий вопрос.
4. В логах видно, почему выбран конкретный маршрут.
5. В неценовых MD-доках отсутствуют числовые цены.

---

## 9) Что нужно от вас (контентная подготовка)

### Обязательно
- Убрать числовые цены из всех неценовых MD-документов.
- Оставить цифры только в:
  - `prices.json`;
  - целевых `*__pricing__*.md` (где действительно нужен нарратив сложной цены).
- Для каждой услуги добавить/проверить YAML-метки и связи:
  - `service_id` (уникальный ID услуги);
  - `doc_role` (`service_entry` / `pricing` / `faq_cost` / `general`);
  - `price_key` (если цена из JSON);
  - `price_ref` (если цена из ценового MD);
  - `concern_ref` (FAQ по дороговизне именно этой услуги).

### Рекомендуется
- Почистить и расширить `aliases` по реальным формулировкам пользователей.
- Проверить, что у каждой ключевой услуги есть:
  - путь для `price_lookup`;
  - путь для `price_concern`;
  - корректный fallback при неуверенном матче.

## 10) Вне рамок текущего шага

- Изменение Python-кода.
- Миграции runtime/инфры.
- Полный рефактор policy.

Этот документ фиксирует целевое ТЗ и требования к данным перед кодовой реализацией.

---

## 11) Свод правил для `service_catalog.json` и `prices.json`

### 11.1 `aliases`
- Только про суть услуги: как ее называют и зачем она нужна.
- Не включать ценовые формулировки (`сколько стоит`, `цена`, `стоимость`).
- Ценовые запросы определяет `classify_intent()`; это отдельный слой.

### 11.2 `facts`
- Только про процедуру и ее плюсы.
- Хранить тезисами, не готовыми фразами (формулировку делает LLM).
- Если факт важен в любом контексте разговора по услуге, он должен быть в `facts`.

### 11.3 `note` в `prices.json`
- Заполнять только при реальном условии, влияющем на цену.
- Примеры: свое КТ влияет на стоимость, скидка при оплате в день обращения, сложность определяется на осмотре.
- Если условия нет, ставить `null` (не заполнять “для вида”).

### 11.4 `facts` vs `note` (разграничение)
- Одна и та же тема может присутствовать в обоих полях, если разный угол:
  - `facts` -> про суть/удобство;
  - `note` -> про денежные условия.
- Дословный дубль между `facts` и `note` не допускается.

### 11.5 `md_entry_ref`
- Хранить без якоря, только `doc_id`.
- Корректно: `clinic__info__tomography`.
- Некорректно: `clinic__info__tomography.md#korotko`.
- Якорь относится к рендеру/навигации, а не к роутингу услуги.

### 11.6 Когда услуга остается в MD, а когда уходит в JSON-card
- Оставлять в MD, если:
  - есть `suggest_h3` с реальным содержательным углублением;
  - есть страхи/возражения/противопоказания;
  - услуга дорогая и требует прогрева.
- Переносить в JSON-card, если:
  - один простой вопрос -> один простой ответ;
  - нет подтем для углубления.

### 11.7 Где живут цены (единое правило)
- Числовые цены только в `prices.json` или в `*__pricing__*.md`.
- Одна цена — одно место хранения (не дублировать).
- `*__pricing__*.md` — исключение для ограниченного числа услуг со сложной пакетной структурой.
- Все остальные MD-документы не содержат числовых цен.

### 11.8 Цена и контент — независимые оси
- Услуга может быть описана в MD или в JSON-card.
- Цена при этом хранится в `prices.json` (кроме согласованных сложных пакетов).
- Изменение цены не требует правок MD.
- Изменение текста услуги не требует правок в ценах.

### 11.9 Необязательный стандарт нейминга (рекомендуется)
- Для упрощения поддержки желательно выравнивать идентификаторы:
  - `service_id` (в каталоге),
  - `subtopic` (в YAML документа),
  - `price_key` (в `prices.json`).
- Строгого технического требования на полное совпадение нет, но единый нейминг снижает риск ошибок при роутинге и ускоряет дебаг.
