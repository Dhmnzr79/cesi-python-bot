# Сценарии и policy-логика стоматологического бота 3.0

Этот файл заменяет черновой `scenario.md` и фиксирует рабочую продуктовую логику бота.

Документ написан сразу в двух режимах:
- **для человека** — чтобы было понятно, как бот должен вести пользователя;
- **для Cursor** — чтобы не осталось двусмысленностей при реализации `policy.py`, `session.py` и `ux_builder.py`.

---

## 1. Главная рамка

Бот нужен **не для длинных консультаций**, а для **мягкого получения лида**.

Это значит:
- бот отвечает по базе знаний;
- не выдумывает факты;
- ведёт по теме последовательно, если тема многошаговая;
- не боится показать CTA чуть раньше, если пользователь уже получил базовую опору;
- может предложить кнопку **«Рассказать о ситуации»** как мягкий шаг к лиду;
- не давит и не обманывает, но и не играет роль «личного друга-консультанта».

**Практический вывод:**
если есть спор между «ещё чуть-чуть пообъяснять» и «мягко подтолкнуть к следующему шагу», бот выбирает следующий шаг, **если экран не перегружен и пользователь уже получил полезный ответ**.

---

## 2. Что считается source of truth

### Контент и маршруты
Источник правды по теме — это `markdown + YAML`.

Именно в документе задаются:
- `doc_id`
- `topic`
- `subtopic`
- `suggest_h3`
- `suggest_refs`
- `cta_text`
- `cta_action`
- `empathy_enabled`
- `situation_allowed`
- `video_key`

### Policy-решения
Решения уровня:
- когда показать CTA,
- когда показать followup,
- когда показать видео,
- когда показать «Рассказать о ситуации»,
- когда перенести `suggest_refs`,

живут **в `policy.py`**, а не в prompt и не во фронте.

## 3. Базовые определения

### 3.1. Что такое тема
Тема = один `md`-документ = один `doc_id`.

Примеры:
- `implantation__faq__pain`
- `implantation__pricing__implants`
- `doctors__doctor__khan`

### 3.2. Что такое `### Коротко`
`### Коротко` — это входной обзорный блок темы.

Он:
- даёт базовый ответ;
- снимает первую неопределённость;
- не считается отдельной подтемой для `covered_h3_ids`.

### 3.3. Что такое внутренняя навигация по теме
Внутреннюю навигацию определяет **только `suggest_h3`**.

Это важное правило.

Если в документе есть H3, но они **не перечислены** в `suggest_h3`, бот **не обязан** строить по ним followup-навигацию.

### 3.4. Что такое `suggest_refs`
`Suggest_refs` — это **мостик в соседнюю тему**, а не продолжение текущей темы.

Пример:
- тема «Как проходит имплантация»
- ref: «Цены на импланты»

Это уже переход в другой документ.

### 3.5. Что такое CTA
CTA — это **только основная кнопка записи**.

Например:
```json
{ "text": "Обсудить на консультации", "action": "lead" }
```

Не считаются CTA:
- followup-кнопки,
- `suggest_refs`,
- видео,
- кнопка «Рассказать о ситуации».

---

## 4. Состояние сессии: как хранить

Нужны **два уровня состояния**:

### 4.1. Глобальное состояние сессии
Используется для общей истории, дебага и технической аналитики.

```python
session = {
    "session_turn_count": 7,
    "current_doc_id": "implantation__faq__pain",
    "lead_intent": "none",
    "situation_pending": False,
    "last_bot_action": "offered_subtopic",
    "last_offer_type": "followup",
    "last_presented_buttons": []
}
```

### 4.2. Состояние по теме
Хранится по `doc_id`.

```python
topic_state = {
    "implantation__faq__pain": {
        "doc_turn_count": 2,
        "covered_h3_ids": ["kakuyu-anesteziyu-ispolzuyut"],
        "video_shown": False,
        "video_pending": False,
        "situation_offered": False,
        "refs_deferred": [],
        "cta_shown": True
    }
}
```

### 4.3. Почему `covered` хранится по `doc_id`
Потому что:
- `doc_id` — стабильный идентификатор темы;
- файл можно переименовать;
- topic/subtopic сами по себе могут быть недостаточно уникальны;
- вся навигация и policy в проекте уже мыслятся на уровне документа.

---

## 5. Какой `turn_count` использовать

Нужно хранить **два счётчика**:

### `session_turn_count`
Глобальный по всей сессии.
Нужен для:
- аналитики,
- логов,
- дебага,
- будущей отчётности.

### `doc_turn_count`
Счётчик внутри конкретной темы.
Именно он используется для policy-решений:
- когда рано/не рано для CTA;
- когда показывать видео;
- когда показывать кнопку «Рассказать о ситуации»;
- когда тема считается достаточно прогретой.

### Жёсткое правило
**Все сценарные пороги считаются по `doc_turn_count`, а не по глобальной сессии.**

Иначе пользователь может сначала спросить контакты, потом врача, потом боль — и кнопка ситуации вылезет просто потому, что это уже третий ход вообще, а не третий ход по этой теме.

---

## 6. Формальный критерий: когда тема исчерпана

### Случай A — `suggest_h3` есть
Тема считается исчерпанной, когда пользователь прошёл все подпункты из `suggest_h3`.

Формула:

```python
topic_exhausted = set(covered_h3_ids) >= set(suggest_h3)
```

### Случай B — `suggest_h3` нет
Тема считается исчерпанной сразу после ответа по `### Коротко`.

Это важно.

Для вашего проекта лучше **не делать fallback на все H3 автоматически**.
Если `suggest_h3` не задан, значит управляемой внутренней навигации в теме нет.

### Что засчитывается как `covered`
Подтема считается покрытой, если:
- пользователь нажал followup-кнопку,
- или задал свободный вопрос,
- и retrieval привёл в H3 текущего документа,
- и этот H3 входит в `suggest_h3`.

### Что не засчитывается как `covered`
- просмотр `### Коротко`;
- вопрос, который привёл только в overview;
- переход в другой документ.

---

## 7. Что происходит при переходе по `suggest_refs`

`Suggest_refs` — это переход в **другую тему**.

Значит:
- старое состояние темы сохраняется;
- covered старой темы не сбрасывается;
- создаётся или продолжается `topic_state` для нового `doc_id`;
- новая тема стартует как новая тема, обычно с `#korotko`.

Иначе говоря:
**`suggest_refs` не продолжают текущую тему, а открывают соседнюю.**

## 8. Lead flow и situation flow

### 8.1. Lead flow
У бота один lead-flow: сбор имени и телефона.

Поле состояния:

```python
lead_intent in {
    "none",
    "collecting_name",
    "collecting_phone",
    "ready",
    "submitted"
}
```

Практическое вычисление:

```python
lead_flow_active = lead_intent in {
    "collecting_name",
    "collecting_phone",
    "ready"
}
```

### 8.2. Situation flow
`Situation_pending` — это **не lead-flow**, а отдельный короткий сценарий.

Логика такая:
1. бот предлагает «Рассказать о ситуации»;
2. пользователь кратко описывает свой случай;
3. описание сохраняется как заметка;
4. дальше запускается тот же глобальный lead-flow: имя → телефон.

### Важно
Нельзя смешивать:
- `lead_intent`
- `situation_pending`

Это два разных механизма.

## 9. Общие правила показа элементов

### 9.1. Followup
Followup-кнопки — это переходы **внутри текущей темы**, только по `suggest_h3`.

Максимум на экране одновременно:
- **2 followup**

### 9.2. CTA
CTA — одна основная кнопка записи.

Максимум на экране одновременно:
- **1 CTA**

### 9.3. Видео
Видео — не просто бонус, а демонстрационная фишка.
Если у темы есть `video_key`, видео должно быть **показано хотя бы один раз в рамках этой темы**, если пользователь не ушёл в лид-воронку раньше.

Поэтому у темы с `video_key` есть два состояния:
- `video_shown = False`
- `video_pending = True`

### Правило
Видео не считается опциональным. Оно считается **обязательным к показу при первом удобном окне интерфейса**.

Видео показывается, если:
- у темы есть `video_key`;
- `video_shown = False`;
- `video_pending = True`;
- не идёт lead-flow;
- не идёт `situation_pending`;
- в этом же ответе не показывается `situation`;
- на экране **не больше 1 followup**.

Видео **временно откладывается**, если:
- на экране уже 2 followup;
- в этом ответе логичнее показать `situation`;
- экран перегружен.

Когда видео откладывается, оно **не отменяется**. Оно остаётся в `video_pending` и должно быть показано в одном из следующих ответов по этой теме, как только исчезнет конфликт по экрану.

### Жёсткое правило
Если у темы есть `video_key`, policy должен попытаться показать видео **раньше `suggest_refs`** и по возможности **до выхода пользователя из темы**.

### 9.4. Кнопка «Рассказать о ситуации»
Это мягкий предлидовый шаг.

Показывается, если:
- `situation_allowed: true`;
- `doc_turn_count >= 3`;
- в этой теме её ещё не предлагали;
- не идёт lead-flow;
- на экране не больше 1 followup.

Не показывается, если:
- на экране 2 followup;
- в этом ответе показывается видео;
- пользователь уже в lead-flow.

### 9.5. `suggest_refs`
Это мостик в соседнюю тему.

Показывается обычно:
- после исчерпания темы,
- или сразу после одноэкранной темы,
- если экран не перегружен,
- и если не выбран более приоритетный тяжёлый элемент.

---

## 10. Приоритеты при конфликте элементов

Когда в одном ответе потенциально можно показать несколько вещей, действует жёсткий приоритет.

### 10.1. Приоритет тяжёлых элементов
```text
situation > video > suggest_refs
```

Но с важной оговоркой:
если у темы есть `video_key` и видео ещё не показано, оно становится **обязательным отложенным элементом**, а не просто необязательной опцией.

### 10.2. Что это означает на практике
- если можно показать `situation`, то в этом ответе `video` не показываем, но оставляем `video_pending = True`;
- если `situation` нельзя, но можно показать `video`, то `suggest_refs` в этом ответе не показываем;
- если ни `situation`, ни `video` не показываются, тогда можно показать `suggest_refs`;
- если `video_pending = True`, не надо терять видео при первой же смене экрана: его нужно показать в следующем подходящем ответе по теме.

### 10.3. Жёсткие запреты
Никогда не показывать вместе:
- `video + situation`
- `suggest_refs + 2 followup`
- `2 CTA`

### 10.4. Особое правило по `situation`, `video` и `suggest_refs`
Если тема завершена и одновременно можно показать:
- `situation`
- `video`
- `suggest_refs`
- `cta`

то действуем так:
1. если `video_shown = False` и экран позволяет, сначала показываем `video + CTA`;
2. если в этот момент логичнее показать `situation`, то показываем `situation + CTA`, но сохраняем `video_pending = True`;
3. `suggest_refs` в обоих случаях переносим в `refs_deferred`.

То есть `suggest_refs` всегда самые отложенные, а видео не должно пропасть только потому, что в этот ход победила `situation`.

---

## 11. Правило CTA для лид-бота

Старое грубое правило вида «не показывать CTA до второго хода» для этого проекта **не подходит**.

Бот лид-ориентированный, значит CTA можно показывать смелее.

### Финальное правило CTA
Показывать CTA, если выполняется хотя бы одно условие:

1. есть явный booking intent;
2. тема исчерпана;
3. `doc_turn_count >= 2` и экран не перегружен.

### Комментарий
Это значит:
- в коротких одноэкранных темах CTA можно показывать сразу;
- в многошаговых темах CTA не обязательно тащить на первый ход, но и не надо прятать слишком долго;
- если выбор спорный, для этого проекта лучше **показать CTA**, чем перестараться с осторожностью.

## 12. Сценарии

Ниже — базовые сценарии, которыми должен руководствоваться policy layer.

---

## Сценарий 1 — одноэкранная тема: только `### Коротко`

### Когда это бывает
Самые простые темы:
- контакты,
- врач,
- короткая общая информация,
- FAQ без внутренней навигации.

### Формальный признак
- `suggest_h3` отсутствует или пустой.

### Ход 1
Бот отвечает по `### Коротко`.

Показываем:
- `suggest_refs` — **0–1**, если есть;
- `CTA` — **да**, потому что тема уже исчерпана.

Не показываем:
- followup;
- видео;
- `situation`.

### Комментарий
Это важный сценарий для продаж.
Если тема закрывается за один ответ, нет смысла искусственно задерживать CTA.
Пользователь уже получил базовую опору — можно сразу дать следующий шаг.

### Типичный пример
- карточка врача;
- тема «как проходит имплантация», если она оформлена одним обзорным блоком и внешним мостиком в цены.

---

## Сценарий 2 — `### Коротко` + 1 подтема

### Когда это бывает
Тема короткая, но требует одного уточняющего шага.

### Формальный признак
- в `suggest_h3` ровно **1** id.

### Ход 1
Бот отвечает по `### Коротко`.

Показываем:
- `followup: [Подтема]`
- `CTA`: обычно **нет**, но допускается, если тема очень коммерческая и экран не выглядит навязчиво.

Не показываем:
- `suggest_refs` — рано;
- `situation` — рано;
- видео — обычно не нужно.

### Ход 2
Пользователь нажал кнопку или задал вопрос, который попал в эту подтему.

Показываем:
- тема считается исчерпанной;
- `suggest_refs` — если есть;
- `CTA` — **да**.

### Комментарий
Это нормальный двухходовый сценарий.
Но для лид-бота допустимо иногда показать CTA и на первом ходе, если тема очень прикладная и followup не критичен.

---

## Сценарий 3 — `### Коротко` + 2–3 подтемы

### Когда это бывает
Основной сценарий для тем типа:
- боль,
- приживаемость,
- цены,
- методы имплантации,
- страховые и возраженческие темы.

### Формальный признак
- в `suggest_h3` 2 или 3 id.

### Ход 1
Бот отвечает по `### Коротко`.

Показываем:
- до **2 followup** из ещё не покрытых подтем;
- CTA — обычно **нет**, но допустимо показать, если тема явно коммерческая и экран не перегружен.

Не показываем:
- `situation` — рано;
- видео — **если на экране уже 2 followup**, видео в этот ход откладываем.

### Комментарий
Это важное уточнение.
Даже если у темы есть `video_key`, на первом экране видео часто лучше **не показывать**, если уже есть две подтемы. Иначе экран становится тяжёлым.
Но видео при этом не отменяется: оно остаётся обязательным к показу на одном из следующих ходов по теме.

### Ход 2
Пользователь открыл одну подтему.

Обновляем:
- `covered_h3_ids += opened_h3`

Показываем:
- только непройденные followup;
- CTA — можно, если `doc_turn_count >= 2` и экран не перегружен;
- видео — нужно показать, если остался максимум 1 followup, видео ещё не показывали и нет конфликта с `situation`.

Не показываем:
- `situation` — обычно ещё рано.

### Ход 3 — если было 2 подтемы
Если все подтемы пройдены:
- тема исчерпана;
- CTA — **да**;
- `situation` — **да**, если разрешена и ещё не предлагалась;
- `suggest_refs` — переносим, если одновременно показываем `situation`.

То есть:
- приоритет: `situation + CTA`
- `suggest_refs` → в `refs_deferred`

### Ход 3 — если было 3 подтемы
Если после второго хода осталась ещё одна подтема:

Показываем:
- `followup: [последняя подтема]`
- `CTA` — **да**, потому что бот уже достаточно прогрел тему и в этом проекте допустима более продающая логика.

Не показываем:
- `situation`, если экран и так занят.

### Ход 4 — для темы с 3 подтемами
Когда последняя подтема пройдена:
- тема исчерпана;
- `CTA` — да;
- `situation` — да, если ещё не показывали;
- `suggest_refs` — либо сразу, если нет `situation`, либо откладываем.

### Комментарий
Это основной рабочий сценарий для стоматологического бота.
Он сочетает:
- полезное раскрытие темы,
- мягкий прогрев,
- переход к лиду без лишнего затягивания.

## 13. Отдельные правила по видео

### Зачем видео вообще нужно
Видео — не декоративный блок, а важная демонстрационная фишка.
Если заказчик добавил `video_key` к теме, значит он ожидает, что видео реально увидят.

### Базовое правило
Для темы с `video_key` действует инвариант:
**видео должно быть показано хотя бы один раз в рамках этой темы**, если пользователь не ушёл в lead-flow раньше, чем экран позволил это сделать.

### Когда видео хорошо работает
- тема болевая или эмоциональная;
- экран не забит кнопками;
- пользователь уже получил базовую структуру ответа;
- видео ещё не показывали.

### Когда видео лучше временно отложить
- на экране уже 2 followup;
- в этом же ответе сильнее смотрится `situation`;
- видео начинает ломать чистоту экрана.

### Практический вывод
Для этого проекта правило не такое: «показать видео, если удобно».
Правило такое: **показать видео обязательно, как только это можно сделать без конфликта по экрану**.

---

## 14. Отдельные правила по кнопке «Рассказать о ситуации»

### Что это такое
Это не просто декоративная кнопка.
Это мягкий переход от общего ответа к персональному кейсу пользователя.

### Почему она важна
Для лид-бота это сильный шаг:
- пользователь чувствует, что его случай готовы услышать;
- это снижает барьер перед записью;
- описание ситуации можно сохранить в lead note.

### Но это не надо пихать слишком рано
Если показать её слишком рано:
- пользователь ещё не получил базовую опору;
- экран будет перегружен;
- кнопка может выглядеть нелогично.

### Поэтому правило такое
Сначала:
- дать ответ;
- при необходимости провести по 1–2 подпунктам;
- затем предложить «Рассказать о ситуации» как мягкий персональный шаг.

## 15. Минимальная матрица экрана

Максимально допустимые наборы:

### Базовый набор
- `followup × 2`
- `CTA × 1`

### С видео
- `followup × 1`
- `video × 1`
- `CTA × 1`

### С ситуацией
- `followup × 1`
- `situation × 1`
- `CTA × 1`

### Финал темы
- `suggest_refs × 1`
- `situation × 1` **или** ничего вместо неё
- `CTA × 1`

### Запреты
Никогда:
- `video + situation`
- `suggest_refs + 2 followup`
- `2 CTA`

## 16. Как policy должен мыслить в спорной ситуации

Если есть сомнение, как поступить, policy должен идти по такому порядку:

1. Не идёт ли сейчас lead-flow?
2. Не идёт ли сейчас situation flow?
3. Какая это тема: одноэкранная или многошаговая?
4. Есть ли ещё непройденные `suggest_h3`?
5. Исчерпана ли тема?
6. Не перегружен ли экран?
7. Можно ли уже показать CTA?
8. Можно ли уже показать `situation`?
9. Не надо ли отложить `suggest_refs`?

### Смысл этого порядка
Сначала бот уважает активный сценарий,
потом — логику темы,
и только потом — дополнительные UI-усилители.

## 17. Что точно не делать

### Не делать так
- не считать все H3 автоматически обязательными подтемами;
- не использовать глобальный `session_turn_count` для сценарных порогов;
- не считать `suggest_refs` продолжением той же темы;
- не смешивать `lead_intent` и `situation_pending`;
- не показывать два тяжёлых элемента сразу;
- не прятать CTA слишком долго «из вежливости».

### Почему
Потому что этот бот — не энциклопедия и не «друг».
Это аккуратный стоматологический продажник с опорой на базу знаний.

## 18. Короткая выжимка для Cursor

Ниже — версия без пояснений, как финальная спецификация.

### Решения
- `topic = one md document = one doc_id`
- `overview = ### Коротко`
- `suggest_h3` — единственный источник внутренней followup-навигации
- если `suggest_h3` нет → тема одноэкранная и считается исчерпанной после `Коротко`
- `covered_h3_ids` хранить по `doc_id`
- `doc_turn_count` использовать для всех сценарных порогов
- `session_turn_count` использовать только для аналитики/дебага
- `suggest_refs` — переход в другой документ, не продолжение текущего
- CTA — только primary booking action
- `lead_intent` и `situation_pending` — разные механизмы
- приоритет тяжёлых элементов: `situation > video > suggest_refs`
- если у темы есть `video_key`, видео становится `video_pending` и должно быть показано хотя бы один раз в теме
- не показывать `video`, если на экране уже 2 followup; в таком случае видео откладывается, а не отменяется
- CTA показывать, если booking intent, или тема исчерпана, или `doc_turn_count >= 2` и экран не перегружен
- если одновременно возможны `situation + suggest_refs + CTA`, показывать `situation + CTA`, а `suggest_refs` откладывать в `refs_deferred`

## 19. Финальный смысл документа

Этот документ нужен, чтобы у бота не было двух крайностей:
- сухой FAQ без доведения до заявки;
- навязчивый продажник без полезного ответа.

Правильный режим — середина:
- сначала полезный ответ,
- потом логичный следующий шаг,
- потом мягкий lead.

Именно так бот остаётся и полезным, и продающим.
