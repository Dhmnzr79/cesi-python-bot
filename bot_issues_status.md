## 🔴 #5 — Flask + SQLite и несколько воркеров
**Статус:** ⚠️ Частично

Что сделано:
- Прод-старт в контейнере переведен на `gunicorn` с одним воркером.
- Зафиксирован запуск: `gunicorn -w 1 -b 0.0.0.0:8000 app:app`.
- Добавлен `gunicorn` в зависимости проекта.

Что остается риском:
- Это гарантирует 1 воркер только для текущего контейнерного пути запуска.
- Если бот поднимут вне этого сценария и зададут `-w 2+`, риск для SQLite-сессий вернется.

Что нужно добить для полного закрытия:
- Зафиксировать единый production entrypoint в репозитории (`Procfile`/`start.sh`) с `-w 1`.
- Добавить короткое правило в деплой-доку: при SQLite-сессиях использовать только 1 воркер.

Файлы:
- `Dockerfile`
- `requirements.txt`

## Короткая фиксация финальной архитектуры

- Бизнес-решения остаются в `policy.py`.
- Контентные настройки задаются через YAML/frontmatter (включая `cta_from_turn`).
- Rerank вызывается выборочно и работает по строгому JSON-контракту.
- При неоднозначном `да` на несколько followup-кнопок бот не принимает решение за пользователя.
- `situation_note` — часть lead-контура, не сигнал для LLM/retrieval.
