## 🔴 #1 — Бот боится продавать (CTA слишком поздно)

**Для тебя:**
Сейчас бот показывает кнопку "Записаться на консультацию" только начиная со второго ответа по теме. То есть: человек спросил про имплантацию, получил ответ — кнопки нет. Только если задаст ещё один вопрос по той же теме — появится. Для продажника это катастрофа: человек может уйти после первого же ответа, так и не увидев CTA.

**Для Cursor:**
В файле `policy.py`, функция `build_policy_decision`, строка:
```python
if show_cta and doc_turn_before < 1:
    show_cta = False
```
Изменить `< 1` на `< 0`. Это уберёт задержку и разрешит CTA с первого ответа по теме.

Дополнительно: сделать этот порог настраиваемым через frontmatter YAML документа. Добавить поле `cta_from_turn: 0` (или `1`) в `meta_loader.py` и передавать его в `build_policy_decision` как параметр. Если поле не указано — дефолт `0`.

## 🔴 #2 — История диалога пишется дважды (сломанный контекст)

**Для тебя:**
Бот запоминает историю разговора, чтобы понимать короткие вопросы вроде "А это дорого?" в контексте предыдущей темы. Но сейчас каждый вопрос пользователя записывается в историю два раза. Это как если бы ты сказал одну фразу, а собеседник записал её в блокнот дважды. Следующий вопрос будет обработан с задвоенным контекстом, что может путать логику переформулировки запросов.

**Для Cursor:**
Конфликт между двумя вызовами `mem_add_user`:

1. В `llm.py`, функция `generate_answer_with_empathy`, первая строка: `mem_add_user(session_id, user_q)` — здесь пишется user-сообщение.
2. В `app.py`, функция `_service_reply`: `mem_add_user(sid, q)` — здесь то же сообщение пишется ещё раз.

Нужно убрать вызов `mem_add_user` из `generate_answer_with_empathy` в `llm.py`. Запись в историю должна происходить только в одном месте — в `_service_reply` в `app.py`. Аналогично проверить `mem_add_bot`: в `generate_answer_with_empathy` есть `mem_add_bot(session_id, answer)`, и в `_service_reply` тоже есть `mem_add_bot(sid, answer)` — убрать дублирование.

## 🔴 #3 — pymorphy3 может не быть в зависимостях

**Для тебя:**
Бот умеет понимать разные формы слов (падежи, склонения) через библиотеку pymorphy3. Без неё "имплантов", "имплантам", "имплантацию" бот будет воспринимать как разные слова. Если эта библиотека не указана в файле зависимостей — на сервере она просто не установится, и бот будет работать хуже без каких-либо ошибок (тихо деградирует).

**Для Cursor:**
Проверить файл `requirements.txt` (или `pyproject.toml`). Убедиться, что там есть строка `pymorphy3`. Если нет — добавить. Также добавить `pymorphy3-dicts-ru` (словари для русского языка, без них библиотека не работает корректно).

## 🔴 #4 — Лиды сохраняются только в файлы на сервере

**Для тебя:**
Когда человек оставляет заявку (имя + телефон), бот сохраняет её в папку `leads/` на сервере в виде json-файлов. Никто об этом автоматически не узнаёт — ни ты, ни администратор клиники. Заявки будут копиться в файлах, о которых никто не будет знать.

**Для Cursor:**
В файле `lead_service.py`, функция `handle_lead`, после строки `json.dump(rec, f, ...)` добавить отправку уведомления. Минимальный вариант — отправка в Telegram через бот:
```python
import requests
tg_token = os.getenv("TELEGRAM_BOT_TOKEN")
tg_chat_id = os.getenv("TELEGRAM_CHAT_ID")
if tg_token and tg_chat_id:
    msg = f"Новая заявка\nИмя: {name}\nТелефон: {phone}\nСитуация: {situation_note}"
    requests.post(f"https://api.telegram.org/bot{tg_token}/sendMessage",
                  json={"chat_id": tg_chat_id, "text": msg}, timeout=5)
```
Добавить `TELEGRAM_BOT_TOKEN` и `TELEGRAM_CHAT_ID` в `.env`. Обернуть в try/except чтобы ошибка отправки не ломала сохранение лида.

## 🔴 #5 — Flask + SQLite не работает нормально с несколькими воркерами

**Для тебя:**
Когда бот запускается на сервере, обычно запускают несколько параллельных процессов (воркеров) для скорости. Но сессии бота хранятся в SQLite — файловой базе данных, которая плохо справляется с несколькими воркерами одновременно. Это может приводить к тому, что бот "забывает" историю разговора или путает сессии разных пользователей.

**Для Cursor:**
При запуске через gunicorn использовать строго одного воркера: `gunicorn -w 1 app:app`. Добавить это явно в `Procfile` или `start.sh`. Это ограничение на текущую версию — в следующей версии нужно мигрировать сессии на Redis или PostgreSQL.

## 🔴 #6 — LLM rerank работает хуже, чем мог бы

**Для тебя:**
Когда бот не уверен, какой из нескольких найденных фрагментов текста лучше подходит для ответа — он просит ещё одну AI-модель выбрать лучший. Но задание этой модели сформулировано очень плохо: нет контекста о том, что это стоматология и что отвечает врач пациенту. Плюс там опечатка ("Ответи" вместо "Ответь"). Это снижает точность выбора.

**Для Cursor:**
В файле `retriever.py`, функция `llm_rerank`:

Исправить опечатку в `prompt`: `"Ответи номером"` → `"Ответь номером"`.

Заменить системный промпт:
```python
# было:
{"role": "system", "content": "Ты выбираешь лучший фрагмент."}
# стало:
{"role": "system", "content": (
    "Ты помощник стоматологической клиники. Тебе нужно выбрать фрагмент из базы знаний, "
    "который наиболее точно отвечает на вопрос пациента. "
    "Отвечай только цифрой: 1, 2 или 3."
)}
```

Также в формировании user-сообщения передавать не только `text`, но и заголовки. Заменить `cands[0]['text'][:600]` на:
```python
f"{cands[0].get('h2','')}\n{cands[0].get('h3','')}\n{cands[0].get('text','')[:500]}"
```
для каждого из трёх кандидатов.

## 🟡 #7 — Короткие реплики после нажатия на followup кнопку

**Для тебя:**
Человек нажал на кнопку "Этапы имплантации" (followup), затем написал "Да" или "Расскажи подробнее". Бот не понимает, что это ответ на предложенную тему, и отправляет "Да" в поиск — откуда ничего хорошего не найдёт. Это частый сценарий, особенно на мобильных устройствах.

**Для Cursor:**
В файле `flow_handlers.py`, функция `handle_flows`, добавить новый блок ПЕРЕД блоком `if is_active_lead_flow(st)`:

```python
# Если последнее действие бота — предложение подтемы (followup кнопки),
# и вопрос очень короткий — считаем это подтверждением и идём по первой кнопке
if (
    st.get("last_bot_action") == "offered_subtopic"
    and q
    and len(q.strip().split()) <= 3
    and parse_yes(q)
):
    buttons = st.get("last_presented_buttons") or []
    if buttons and buttons[0].get("ref"):
        first_ref = buttons[0]["ref"]
        # Вернуть сигнал для app.py чтобы он пошёл по ref-ветке
        return {"payload": None, "doc_id": None, "redirect_ref": first_ref}
```
В `app.py` обработать `redirect_ref` в результате `handle_flows` — идти по ветке `ref`.

## 🟡 #8 — "Ситуация" не передаётся в LLM при генерации ответа

**Для тебя:**
Когда человек описывает свою ситуацию ("у меня диабет, боюсь имплантации"), бот сохраняет этот текст и потом отправляет его вместе с лидом администратору. Но при генерации ответа LLM не знает об этой ситуации — то есть ответ не учитывает, что человек рассказал о себе. Слабое место для эмпатичного продажника.

**Для Cursor:**
В файле `chunk_responder.py`, функция `respond_from_chunk`, перед вызовом `generate_answer_with_empathy` добавить получение situation_note из сессии:

```python

st_for_situation = mem_get(sid)
situation_note = (st_for_situation.get("situation_note") or "").strip()
```

В `llm.py`, функция `build_messages_for_gpt`, в user-сообщение добавить situation_note если он непустой:

```python
situation_block = f"\nСитуация пациента: {situation_note}\n" if situation_note else ""
content = (
    "Вопрос пациента:\n"
    + user_q.strip()
    + situation_block
    + "\n\nКонтент для ответа...\n"
    + context_md.strip()
)
```
Добавить `situation_note: str = ""` как параметр в `generate_answer_with_empathy` и `build_messages_for_gpt`.

YES_RX = re.compile(
